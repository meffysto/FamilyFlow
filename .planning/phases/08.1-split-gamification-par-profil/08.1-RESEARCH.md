# Phase 08.1: Split Gamification Par Profil - Research

**Researched:** 2026-03-30
**Domain:** File architecture migration — gamification.md → gami-{id}.md per-profile files
**Confidence:** HIGH

## Summary

The current architecture centralises all profile gamification state (points, level, streak, loot boxes, history, active rewards, used loots) in a single `gamification.md` file. All hooks and vault utilities read and write this file directly by its hardcoded name. On multi-device iCloud setups this creates write conflicts: two devices writing the same file in close succession produce a corrupted or partial save because iCloud's `NSFileCoordinator` cannot merge simultaneous writes to a single file.

The solution is to split `gamification.md` into one file per profile: `gami-{profileId}.md`. Each profile owns its data independently — reads and writes are scoped to a single file that only one device (the one belonging to that profile's user) ever writes. History, active rewards, and used loots are all scoped to a single profile, which is consistent with the existing `profileId` field on every `GamificationEntry`, `ActiveReward`, and `UsedLoot`.

The migration must be backward-compatible: on first load after the update, if the old `gamification.md` exists the app must split it into per-profile files and then leave `gamification.md` in place (or rename it) so data is never lost. The in-memory `GamificationData` type that the rest of the app consumes via `gamiData` from `useVault()` is a merged view across all per-profile files — the public API stays the same.

**Primary recommendation:** Replace the single `GAMI_FILE = 'gamification.md'` constant with a `gamiFile(profileId)` helper returning `gami-${profileId}.md`. All read/write call sites pass the profile ID. On app load, `loadVaultData` migrates if needed. The `GamificationData` type and all consumer screens are left unchanged.

## Architecture Patterns

### Current File Layout
```
vault/
├── famille.md           # profile metadata (name, role, avatar, ...)
├── gamification.md      # ALL profiles: points, level, history, rewards, usedLoots
└── ...
```

### Target File Layout
```
vault/
├── famille.md           # unchanged
├── gamification.md      # KEPT during migration (renamed to .bak or left as-is)
├── gami-papa.md         # one file per profile ID
├── gami-maman.md
├── gami-lucas.md
└── gami-emma.md
```

### Pattern: Per-Profile File Path Helper
**What:** A small pure helper that maps a profile ID to its file path.
**When to use:** Every read/write call site replaces `GAMI_FILE` with `gamiFile(profileId)`.
**Example:**
```typescript
// lib/parser.ts or hooks/useVault.ts
export function gamiFile(profileId: string): string {
  return `gami-${profileId}.md`;
}
```

### Pattern: Single-Profile Parse / Serialize
**What:** `parseGamification` today parses multiple `## Name` sections. After split, each file contains exactly one profile's data.
**When to use:** Every per-profile file read.

The current parser already supports single-profile files without modification — when a file contains only one `## Name` section it returns `profiles: [single]`. No parser change is strictly required. However, the serialize path should also remain identical.

**Verification:** `parseGamification(singleProfileContent).profiles` returns exactly one entry.

### Pattern: Merged In-Memory View (unchanged public API)
**What:** `loadVaultData` reads all per-profile gami files and merges them into a single `GamificationData` to populate `gamiData` state. Consumer screens (`loot.tsx`, `tree.tsx`, `index.tsx`) are unchanged.
**When to use:** During initial load and `refresh()`.
```typescript
// Inside loadVaultData — replace single-file read with multi-file read
const gamiContents = await Promise.all(
  baseProfiles.map(p =>
    vault.readFile(gamiFile(p.id)).catch(() => '')
  )
);
// Merge all per-profile data into one GamificationData
const allGami = gamiContents.map(c => parseGamification(c));
const merged: GamificationData = {
  profiles: allGami.flatMap(g => g.profiles),
  history: allGami.flatMap(g => g.history),
  activeRewards: allGami.flatMap(g => g.activeRewards ?? []),
  usedLoots: allGami.flatMap(g => g.usedLoots ?? []),
};
```

### Pattern: Scoped Write (per profile)
**What:** Every write operation that currently touches `gamification.md` must now identify the target profile(s) and write only to that profile's file.
**When to use:** All mutation callbacks.

The key constraint: `history`, `activeRewards`, and `usedLoots` are already tagged with `profileId` on every record — scoping writes to a per-profile file is always correct.

```typescript
// Before (all hooks):
const gamiContent = await vault.readFile(GAMI_FILE);
const gami = parseGamification(gamiContent);
// ... mutate ...
await vault.writeFile(GAMI_FILE, serializeGamification(gami));

// After:
const file = gamiFile(profileId);
const gamiContent = await vault.readFile(file).catch(() => '');
const gami = parseGamification(gamiContent);
// ... mutate single-profile data ...
await vault.writeFile(file, serializeGamification(singleProfileData));
```

### Pattern: One-Time Migration on Load
**What:** On `loadVaultData`, if `gamification.md` exists and any `gami-{id}.md` does NOT exist, split the old file into per-profile files.
**When to use:** `loadVaultData` only (runs on startup and refresh).
**Migration logic:**
```typescript
async function migrateGamification(vault: VaultManager, profiles: BaseProfile[]): Promise<void> {
  const legacyExists = await vault.exists('gamification.md');
  if (!legacyExists) return;
  const legacyContent = await vault.readFile('gamification.md');
  const legacyData = parseGamification(legacyContent);
  for (const prof of legacyData.profiles) {
    const file = gamiFile(prof.id);
    const alreadyMigrated = await vault.exists(file);
    if (!alreadyMigrated) {
      const singleData: GamificationData = {
        profiles: [prof],
        history: legacyData.history.filter(e => e.profileId === prof.id),
        activeRewards: (legacyData.activeRewards ?? []).filter(r => r.profileId === prof.id),
        usedLoots: (legacyData.usedLoots ?? []).filter(u => u.profileId === prof.id),
      };
      await vault.writeFile(file, serializeGamification(singleData));
    }
  }
  // Legacy file left intact — safe to remove in a future phase
}
```

Note: Do NOT delete `gamification.md` in this phase. Removing it after migration is a separate opt-in step to protect against migration bugs. Keep it as a read-only backup.

### Anti-Patterns to Avoid
- **Reading GAMI_FILE after migration:** Any code that still reads `gamification.md` after migration will see stale data. Must audit every `GAMI_FILE` reference and replace.
- **Writing merged GamificationData to all profile files:** A write that reads `gamiData` (the merged state) and serializes it to a per-profile file will write other profiles' history into the wrong file. Always filter by `profileId` before writing.
- **Forgetting `gamiData` in-memory on mutations:** After writing to `gami-{id}.md`, update `gamiData` state with a partial merge (replace just that profile's slice), not a full disk reload. This keeps the UI reactive.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Per-file write serialisation | Custom async lock/mutex | `VaultManager._writeQueues` (already keyed by path) | Already implemented and battle-tested in Phase 2 |
| iCloud coordinated writes | Direct FileSystem.writeAsync | `VaultManager.writeFile` (uses coordinatedWriteFile) | NSFileCoordinator already integrated |
| Profile ID normalisation | Inline `.toLowerCase().replace(...)` | Use the existing `profile.id` value from `parseFamille` | IDs are already normalized at parse time |

**Key insight:** The write queue in `VaultManager` is already keyed by relative path. Splitting into per-profile files means per-profile queues are automatically created — concurrent writes for different profiles no longer block each other, and concurrent writes for the same profile are still serialised.

## Common Pitfalls

### Pitfall 1: Forgetting useFarm.ts
**What goes wrong:** `hooks/useFarm.ts` defines its own `GAMI_FILE = 'gamification.md'` constant (line 42). It reads and writes gamification independently from `useGamification` and `useVault`. If left unchanged it will write to the old file after migration.
**Why it happens:** Two separate constants define the same path string. Easy to miss in audit.
**How to avoid:** Search for `GAMI_FILE` in all `hooks/*.ts` — update `useFarm.ts` separately. The `deductLeaves` and `addLeaves` functions must accept a `profileId` parameter.
**Warning signs:** Farm leaves not updating after migration.

### Pitfall 2: addChild / initVault in vault.ts still writes gamification.md
**What goes wrong:** `VaultManager.addChild` (line 601-611) and `initVault` (line 405-414) both write directly to `gamification.md`. After migration, new profiles added via `addChild` will create a `gamification.md` entry instead of a `gami-{id}.md` file.
**Why it happens:** `vault.ts` is not a hook — it doesn't use the `GAMI_FILE` constant from hooks; it uses the literal string `'gamification.md'`.
**How to avoid:** Update `addChild` to write to `gamiFile(id)` and update `initVault` to write per-profile files from the start.
**Warning signs:** New children added post-migration have no gamification file.

### Pitfall 3: updateProfile rename logic broken
**What goes wrong:** `updateProfile` in `useVault.ts` (line 1447-1470) renames the `## Name` header inside `gamification.md` when a profile is renamed. With split files the file is named by `profileId` (stable, derived from the original name at creation), and the `## Name` header inside is what changes. But if the user changes their name the `profileId` does NOT change — so the file name is still `gami-{originalId}.md`. The rename logic needs to update only the `## Name` header inside the per-profile file, not the filename.
**Why it happens:** The two concepts (file name = id, section header = display name) are conflated.
**How to avoid:** After migration, the per-profile file is always `gami-{profileId}.md` (ID = stable). The rename only changes the `## Name` line inside the file. File is NOT renamed.
**Warning signs:** TS error or incorrect header after profile rename.

### Pitfall 4: mergeProfiles signature change must be backward-compatible
**What goes wrong:** `mergeProfiles(familleContent, gamiContent)` is called in 8+ places across hooks and vault.ts. Changing its signature to accept `gamiContents: string[]` or a `Map` breaks every call site.
**Why it happens:** Broad usage across the codebase.
**How to avoid:** Either keep `mergeProfiles` as-is and pre-merge the GamificationData before calling it, or add an overloaded helper. The simplest approach: merge all per-profile GamificationData in `loadVaultData` into a single `gamiContent` string that the existing `mergeProfiles` can consume. Alternatively, pass the pre-merged `GamificationData` directly to `setProfiles` and bypass `mergeProfiles` in the reload path.
**Warning signs:** TypeScript errors at call sites after refactor.

### Pitfall 5: refreshGamification only reads one file
**What goes wrong:** `refreshGamification` in `useVault.ts` (line 2287) currently reads `GAMI_FILE` directly. If updated to read only the active profile's file, `gamiData` will be incomplete (missing other profiles' history/rewards).
**Why it happens:** The function was simple before; the per-profile change makes it need to read N files.
**How to avoid:** `refreshGamification` must read all profiles' files and merge them, same as `loadVaultData`.
**Warning signs:** Loot box count showing 0 for non-active profiles after refresh.

### Pitfall 6: settings.tsx SettingsGamification reset
**What goes wrong:** The gamification reset in settings (via `SettingsGamification`) likely writes to `gamification.md` directly. Must be updated to reset each profile's per-profile file.
**How to avoid:** Audit `components/` for any `writeFile('gamification.md', ...)` or `writeFile(GAMI_FILE, ...)` usage.

### Merge on load
```typescript
// In loadVaultData, replace the single GAMI_FILE read with:
const gamiFiles = await Promise.allSettled(
  baseProfiles.map(p => vault.readFile(gamiFile(p.id)))
);
const mergedProfiles: Profile[] = [];
const mergedHistory: GamificationEntry[] = [];
const mergedActiveRewards: ActiveReward[] = [];
const mergedUsedLoots: UsedLoot[] = [];
for (let i = 0; i < baseProfiles.length; i++) {
  const result = gamiFiles[i];
  const content = result.status === 'fulfilled' ? result.value : '';
  const gami = parseGamification(content);
  mergedProfiles.push(...gami.profiles);
  mergedHistory.push(...gami.history);
  mergedActiveRewards.push(...(gami.activeRewards ?? []));
  mergedUsedLoots.push(...(gami.usedLoots ?? []));
}
const gamiData: GamificationData = {
  profiles: mergedProfiles,
  history: mergedHistory,
  activeRewards: mergedActiveRewards,
  usedLoots: mergedUsedLoots,
};
```

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| Single gamification.md for all profiles | Per-profile gami-{id}.md | Phase 08.1 | Eliminates iCloud write conflicts; each device writes only its own profile file |

**The iCloud conflict model:**
- iCloud Drive uses `NSFileCoordinator` for coordinated access. Two devices writing `gamification.md` in close succession result in the "conflict version" mechanism — iCloud keeps both but the app may read either one.
- With per-profile files, Device A (Papa) writes `gami-papa.md` and Device B (Maman) writes `gami-maman.md` — no overlap, no conflict.

## Open Questions

1. **Should gamification.md be deleted after migration?**
   - What we know: Migration writes new per-profile files. Legacy file is then stale.
   - What's unclear: If the user has two devices and only one is updated, the old device still writes `gamification.md`. Deleting it prematurely could lose data.
   - Recommendation: Leave `gamification.md` intact in this phase. Document a future cleanup phase. The migrated per-profile files take precedence.

2. **How to handle the global sections (history, activeRewards, usedLoots) that contain mixed-profile entries?**
   - What we know: Every entry has a `profileId` field — filtering is safe and complete.
   - What's unclear: Nothing — filtering is the correct approach.
   - Recommendation: Filter by `profileId` when writing per-profile files. No entries will be lost.

3. **Does completeAdventure need to handle multiple profiles?**
   - What we know: Line 3084 uses the hardcoded string `'gamification.md'` (not the constant). It receives `childProfileId` as a parameter.
   - Recommendation: Replace with `gamiFile(childProfileId)` and scope the read/write to that profile.
