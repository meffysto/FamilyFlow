## 1. System Classification

**System Type:** Content Generation Pipeline + Quality Gating (hybrid : génération existante + évaluation déterministe + LLM-judge async)

**Description:**
Pipeline qui score chaque histoire du soir produite par `generateBedtimeStory()` (lib/ai-service.ts:992) sur 6+ dimensions calibrées (longueur, fin paisible, vocabulaire varié, anti-clones, cohérence saga, tags TTS), bloque les hard fails avec re-roll automatique (cap 1 retry, prompt augmenté avec issues), et propose un LLM-eval async post-show pour la qualité narrative subjective. Utilisé par les parents (single-user familial) pour assurer une qualité constante des histoires bedtime quotidiennes générées par Claude Sonnet via API directe. "Good" = pipeline transparent, score visible non-intrusif, hard fails attrapés avant que l'enfant n'écoute, distribution flagged stable vs golden set baseline (14/20 attendus).

**Critical Failure Modes:**
1. **Régression silencieuse du rubric** — un changement de seuil ou de helper fait flag soudain 100% ou 0% des stories, détectable uniquement avec golden set test de régression (20 histoires hand-rated dans `STORIES-NOTABLE.md`)
2. **Re-roll qui dégrade** — la 2ème génération est encore pire que la 1ère (rare mais possible si les "issues" injectées dans le prompt déstabilisent Claude). Mitigation : cap 1 retry max, fallback ship 1ère version si 2ème échoue aussi
3. **LLM-eval qui hallucine** — Claude Haiku peut inventer des dimensions, retourner des scores hors-range, ou produire du JSON malformé. Mitigation : Zod schema validation + clamping 0-10 + logging __DEV__
4. **Persistence frontmatter casse le parser existant** — l'ajout de `quality_score` / `quality_issues` dans le frontmatter doit roundtrip propre via `parseStoryFrontmatter` (lib/parser.ts:3591) et `serializeBedtimeStory` (lib/parser.ts:3539). Test obligatoire : créer story → save → re-read → fields égaux
5. **Coût LLM-eval qui explose** — appel synchrone bloquant l'UX bedtime ou retry en boucle = parent en attente + facture API. Mitigation : LLM-eval **toujours async post-show**, jamais sur le chemin critique de génération + cap 1 retry sur déterministe uniquement

## 1b. Domain Context

> Researched by `gsd-domain-researcher`. Grounds the evaluation strategy in domain expert knowledge.

**Industry Vertical:** Hybride **littérature jeunesse 3-6 ans** (canon École des Loisirs / Flammarion Jeunesse / albums illustrés) + **sleep coaching enfant** (psychologie de l'endormissement, rituel du coucher). Pas de vertical "AI bedtime stories" mature à ce jour — la baseline reste l'album papier lu à voix haute par un parent.

**User Population:** Parents (single-user familial) qui lisent l'histoire à l'enfant 3-9 ans avant le coucher. Contexte : moment câlin, enfant fatigué, faible tolérance à la frustration, attention de 5-15 min. Le parent veut une histoire (a) facile à lire à voix haute, (b) qui calme et n'excite pas, (c) qui personnalise sans saturer, (d) renouvelée mais cohérente avec les précédentes (effet série).

**Stakes Level:** **Medium** — pas vital mais répété quotidiennement, donc impact accumulé. Une histoire ratée isolée est tolérable ("on en relit une autre"), mais un pattern destructeur (saturation, vocab trop abstrait, fin agitée) érode la confiance dans l'app et perturbe le rituel coucher. Pas Low car l'enfant peut littéralement perdre le sommeil sur une histoire mal calibrée ; pas High car aucun risque physique/santé direct.

**Output Consequence:** Si l'histoire est mauvaise → (1) parent ressent l'échec et perd confiance dans la feature, (2) enfant s'agite au lieu de s'endormir (échec direct du rituel), (3) frustration redirige l'usage vers livres papier ou apps concurrentes, (4) accumulation = désengagement progressif. Pas de risque réglementaire ou contentieux direct, mais perte de churn qualitative pour l'écosystème app.

### Known Failure Modes in This Domain

Mappés 1:1 sur les patterns P1-P7 du baseline `ANALYSIS.md` (audit 20 histoires) :

1. **Stimulant en fin** → cliffhanger, exclamation, action en dernier paragraphe → enfant agité au lit. Observé 0/20 dans le baseline (le prompt actuel sait gérer ce cas) mais reste un risque structurel.
2. **Vocabulaire abstrait pour 3-5 ans** → mots comme "courage", "valeureux", "fierté" sans ancrage concret → enfant décroche, parent explique → casse le rythme. Observé partiellement (#5, #14 — "courageux" sans contexte concret).
3. **Cliché à répétition** ↔ **P2 "doucement" 4-8x/histoire** → "doucement", "tout doucement, tout doucement", "Un pas. Deux pas. Trois pas." formule en boucle. Observé **14/20**, pattern dominant.
4. **Tags TTS leakés** ↔ **P1** → `[whispers]`, `[chuckles]` rendus dans le texte → lus à voix haute par le parent ou par expo-speech. Observé **18/20** (résolu côté code par strip post-génération depuis commit `30ab12d1`).
5. **Saturation thématique** ↔ **P3 "premières-fois marche seul"** → 15+ histoires consécutives sur le même pivot narratif → perte de signification + lassitude enfant. Observé **18/20**.
6. **Entités nommées incohérentes** ↔ **P4 "Cagou/Cacou"** → un mot d'enfant (Cagou) redéfini dans chaque univers (caillou, dragon, étoile, fleur) → confusion cognitive 3-5 ans. Observé **8/20**.
7. **Titres dupliqués + quasi-clones** ↔ **P5+P7** → "Le premier pas dans la forêt" 5× sur 2 jours, paragraphes 70% identiques → effet "déjà-vu" parent/enfant. Observé **5/20**.
8. **Typos lexicales / corruption morphosyntaxique** ↔ **P6** → "papates rondes", "museau dessert" → casse la lecture à voix haute, demande explication. Observé **2/20** (rare mais aucun garde-fou actuel).

## 2. Framework Decision

**Selected Framework:** No framework — pure TypeScript + direct Anthropic SDK (already wired) + Jest + optional Zod for LLM-eval JSON validation

**Version:**
- TypeScript ~5.x (existing project version)
- `@anthropic-ai/sdk` : NON utilisé — appels directs `fetch()` à `api.anthropic.com/v1/messages` (pattern existant `lib/ai-service.ts`)
- Jest (déjà installé, version projet)
- Zod : à ajouter UNIQUEMENT si validation JSON LLM-eval nécessaire — `^3.x` (lightweight, no codegen)

**Rationale:**
Le rubric est de la logique métier custom calibrée sur un golden set de 20 histoires (patterns P1-P7 avec seuils numériques précis dans `ANALYSIS.md`). Aucun framework générique n'encode ces règles. Le LLM-eval est un seul appel async Claude Haiku qui s'intègre dans le pattern direct-API existant (`lib/ai-service.ts:992` `generateBedtimeStory` utilise déjà `fetch()` direct). Les contraintes dures éliminent toute autre option : RN client-side bundle (pas de Python/serveur), zero-backend (pas de LangGraph), TypeScript strict, persistence vault Markdown (pas de DB). Ajouter un orchestration framework violerait l'Anti-Pattern #1 du decision matrix ("LangChain pour cas simples — direct SDK = moins de code").

**Alternatives Considered:**

| Framework | Ruled Out Because |
|-----------|------------------|
| LangChain.js / LangGraph | Stateful, optimisé pour orchestration multi-step, surdimensionné pour 1 appel LLM-eval async ; abstraction tax sans levier ici |
| Promptfoo | Excellent comme dev-only regression harness mais offline tool ; on garde l'option si Jest golden-set tests deviennent unwieldy (devDependency, jamais dans le bundle RN) |
| Inspect AI (UK AISI) | Python uniquement, élimine d'office (pas de backend Python dans la stack) |
| Braintrust / Langfuse / Phoenix | Plateformes observability serveur/SaaS, violent zero-backend constraint ; pour solo dev + ~1-2 stories/jour, frontmatter persistent + console __DEV__ suffisent |
| RAGAS / Giskard | RAG-focused ou ML-focused, ne couvrent pas content generation rubric |
| Mastra / Vercel AI SDK | Abstraction au-dessus d'Anthropic SDK, ajoute couche sans bénéfice (déjà appel direct fetch) |

**Vendor Lock-In Accepted:** Partial — couplage volontaire à Anthropic API (Claude Sonnet pour génération, Claude Haiku pour LLM-eval) déjà acquis sur la feature Stories. Pas de couplage framework. Le rubric déterministe est portable (TypeScript pur). Le LLM-eval pourrait être migré vers OpenAI / autre en changeant 10 lignes — coût migration faible.

### Entry Point Pattern

Pattern minimal pour une évaluation complète — déterministe synchrone + LLM-judge fire-and-forget. À placer dans `lib/eval/pipeline.ts`.

```typescript
// lib/eval/pipeline.ts
import type { BedtimeStory, ChildProfile } from '../parser';
import type { AIConfig } from '../ai-service';
import { evaluateStoryDeterministic, type RubricResult } from './rubric';
import { evaluateStoryWithLlm } from './llm-eval';

export interface PipelineContext {
  story: BedtimeStory;
  child: ChildProfile;
  recentStories: BedtimeStory[]; // 5 dernières du même enfant pour anti-clones
  aiConfig: AIConfig;
}

/**
 * Évalue une story qui vient d'être générée.
 * - Phase 1 (sync, bloquante) : rubric déterministe → décision re-roll
 * - Phase 2 (async, fire-and-forget) : LLM-judge → patch frontmatter
 *
 * Retourne le résultat déterministe immédiatement.
 * La promesse LLM est exposée pour permettre aux tests d'await,
 * mais l'UI ne DOIT JAMAIS l'attendre sur le chemin critique bedtime.
 */
export function runEvalPipeline(ctx: PipelineContext): {
  rubric: RubricResult;
  llmJudgeBackground: Promise<void>;
} {
  // 1. Déterministe — quelques ms, jamais d'erreur réseau
  const rubric = evaluateStoryDeterministic(ctx.story, ctx.child, ctx.recentStories);

  // 2. LLM-judge en background — n'attend PAS la résolution.
  //    L'UI affiche la story tout de suite ; le score qualitatif arrive plus tard.
  const llmJudgeBackground = evaluateStoryWithLlm(ctx.story, ctx.child, ctx.aiConfig)
    .then((_llmResult) => {
      // L'appelant (hooks/useStories) écrit le frontmatter via VaultManager.
      // Cette couche reste pure : retourne juste les données.
    })
    .catch((e) => {
      if (__DEV__) console.warn('[eval] LLM-judge failed silently:', e);
    });

  return { rubric, llmJudgeBackground };
}
```

### Key Abstractions

| Concept | What It Is | When You Use It |
|---------|-----------|-----------------|
| **Rubric déterministe** | Fonction pure `(story, child, recent[]) => RubricResult` qui calcule scores 1-3 par dimension à partir de regex / comptages / n-grams. Zéro I/O, zéro API. | Sur le chemin critique post-génération, avant `saveStory()`. Décide du re-roll. |
| **Hard fail vs soft warning** | Hard fail = `score===1` sur ≥1 dim critique → re-roll obligatoire (cap 1). Soft warning = `score===2` → loggé en frontmatter, pas de re-roll. | Boucle `generateAndEvaluate()` dans `hooks/useStories`. |
| **LLM-judge async** | 1 appel Haiku 4.5 fire-and-forget après que la story est sauvegardée. Retourne 4 sous-scores narratifs (rythme, originalité, charge émotionnelle, fluidité). | Hors chemin critique. Update frontmatter quand resolve, jamais bloquant. |
| **Golden set fixture** | 20 stories du dataset baseline (`STORIES-NOTABLE.md`) sérialisées en JSON Jest fixture, avec verdict humain attendu (🟢/🟡/🔴). | Test de non-régression du rubric à chaque CI. Doit reproduire 14/20 flagged. |
| **Anonymisation pré-LLM** | Mapping `enfant_name → "Enfant 1"` (réutilise le pattern existant `prepareAnonymized` lib/ai-service.ts:333) avant envoi au judge. | À chaque appel LLM-judge — privacy-by-default. |

### Common Pitfalls

1. **Bloquer l'UX bedtime sur le LLM-judge** — Si on `await` le Haiku call sur le chemin critique, le parent attend 1-2s supplémentaires devant un enfant fatigué. *Symptôme* : feedback "ça lag depuis Phase 52". *Fix* : `runEvalPipeline()` retourne `{ rubric, llmJudgeBackground }`. L'appelant n'await JAMAIS `llmJudgeBackground` dans le flow génération.
2. **Re-roll boucle infinie** — Un cap 1 retry implicite via flag booléen est facile à régresser (un dev rajoute un cas). *Symptôme* : facture API qui grimpe. *Fix* : signature explicite `generateAndEvaluate(ctx, { retriesRemaining: 1 })`, le compteur descend à 0 dans la branche re-roll, branche else garantie de ship.
3. **Frontmatter shape drift casse le cache** — Ajouter `quality_score` au frontmatter sans bumper `CACHE_VERSION` (lib/vault-cache.ts:53) → app re-launch lit du JSON snapshot stale qui n'a pas le champ → crash silencieux dans le dashboard. *Fix* : bumper `CACHE_VERSION` dans la même PR que le changement parser, OU exclure stories du cache (recalcul gratuit au boot via parser).
4. **JSON LLM malformé sans Zod** — Haiku répond parfois avec ` ```json\n{...}\n``` ` (markdown fence) ou injecte du commentaire. *Symptôme* : `JSON.parse()` throw à chaque histoire pendant 1 semaine, pas vu car catché silencieusement. *Fix* : strip fences avant parse + Zod schema + 1 retry max avec prompt durci, sinon score neutre 5/10.
5. **N-gram overlap sur stories anonymisées vs historique non-anonymisé** — Si on compare la story générée (contient "Lucas") avec les `recentStories` anonymisés (contiennent "Enfant 1"), les n-grams divergent artificiellement. *Fix* : passer les deux côtés dans la même normalisation (lowercase + strip prénoms enfants connus du profil) avant comptage.
6. **Tests Jest qui font des vrais appels API** — Le golden set doit être 100% offline. Le rubric déterministe est testable sans réseau ; le LLM-judge doit être moqué (`jest.mock('./llm-eval')`). *Fix* : injecter `evaluateStoryWithLlm` comme dépendance ou utiliser `jest.fn().mockResolvedValue(stubResult)` dans les tests d'intégration.

## 4. Implementation Guidance

**Model Configuration:**

Deux modèles, deux usages — séparation stricte entre génération (existant) et évaluation (nouveau).

| Usage | Modèle | `temperature` | `max_tokens` | Bloquant ? |
|-------|--------|---------------|--------------|------------|
| Génération histoire (existant, inchangé) | `claude-sonnet-4-5-20250929` ou modèle déjà configuré dans `AIConfig.model` | (existant, ~0.8) | (existant, ~1024-2048 selon longueur) | Oui — UX bedtime |
| LLM-judge qualité narrative (nouveau) | `claude-haiku-4-5-20251001` | `0` (reproductibilité — un même texte doit donner ~même score) | `300` (réponse JSON courte ≤ 4 sous-scores + 1 ligne justification) | **Non — async post-show** |

Justification du choix Haiku pour le judge : la tâche est de la lecture critique courte (1500 tokens d'input, ~200 d'output). Haiku 4.5 est ~5× moins cher que Sonnet, latence ~1-2s, et la qualité narrative subjective ne nécessite pas le raisonnement Sonnet. Si on observe (via golden set test) une divergence > 1 point entre Haiku-judge et notation humaine, on bascule sur Sonnet (10 lignes à changer).

**Core Pattern:**

Rubric déterministe — fonctions pures, composables, testables sans réseau.

```typescript
// lib/eval/rubric.ts
import type { BedtimeStory, ChildProfile } from '../parser';
import { STORY_LENGTHS } from '../stories';
import { typeTokenRatio, ngramOverlap, countOccurrences } from './rubric-helpers';

export type DimensionScore = 1 | 2 | 3; // 1 = hard fail, 2 = soft warning, 3 = OK

export interface RubricDimension {
  id: 'longueur' | 'fin_paisible' | 'vocabulaire' | 'anti_clones' | 'tags_tts' | 'coherence_saga';
  score: DimensionScore;
  reason?: string; // français, injectable dans prompt re-roll
}

export interface RubricResult {
  dimensions: RubricDimension[];
  hardFail: boolean;        // ≥1 dimension à 1 sur dim critique → re-roll
  softWarnings: string[];   // raisons des dim à 2 — loggées au frontmatter
  rerollPromptHint: string; // texte FR à injecter dans le prompt si re-roll
}

const CLICHE_FORMULAS: RegExp[] = [
  /tout doucement, tout doucement/i,
  /un pas\.\s*deux pas\.\s*trois pas\./i,
  /petits pieds dans l['']herbe/i,
];

/**
 * Évalue une story en pure synchrone — pas d'I/O, pas de réseau.
 * Doit reproduire (en CI Jest) le verdict humain du golden set à ±1 dim près.
 */
export function evaluateStoryDeterministic(
  story: BedtimeStory,
  _child: ChildProfile,
  recentStories: BedtimeStory[],
): RubricResult {
  const dims: RubricDimension[] = [];

  // D1 — Longueur (cf. ANALYSIS.md seuils §122)
  const target = STORY_LENGTHS[story.length ?? 'moyenne'].targetWords;
  const actual = story.texte.split(/\s+/).filter(Boolean).length;
  const deviation = Math.abs(actual - target) / target;
  dims.push({
    id: 'longueur',
    score: deviation > 0.35 ? 1 : deviation > 0.20 ? 2 : 3,
    reason: deviation > 0.20
      ? `Longueur ${actual} mots vs cible ${target} (écart ${Math.round(deviation * 100)}%)`
      : undefined,
  });

  // D2 — Fin paisible (dernier paragraphe contient marqueur apaisant)
  const lastPara = story.texte.split(/\n\n+/).slice(-1)[0] ?? '';
  const hasCalm = /(s['']endor|ferme.*yeux|paisible|sommeil|rêve)/i.test(lastPara);
  const hasAgitation = /(crie|cours|peur|frayeur|monstre)/i.test(lastPara);
  dims.push({
    id: 'fin_paisible',
    score: hasCalm && !hasAgitation ? 3 : 1,
    reason: !hasCalm
      ? 'Fin pas clairement apaisée'
      : hasAgitation ? 'Marqueur d\'agitation dans la fin' : undefined,
  });

  // D5 — Vocabulaire (type-token + occurrences "doucement" + clichés)
  const ttr = typeTokenRatio(story.texte);
  const doucementCount = countOccurrences(story.texte, /\bdoucement\b/gi);
  const clicheHits = CLICHE_FORMULAS.filter((re) => re.test(story.texte)).length;
  const vocabFail = ttr < 0.40 || doucementCount > 7 || clicheHits >= 2;
  const vocabWarn = ttr < 0.45 || doucementCount > 4 || clicheHits >= 1;
  dims.push({
    id: 'vocabulaire',
    score: vocabFail ? 1 : vocabWarn ? 2 : 3,
    reason: vocabFail || vocabWarn
      ? `TTR ${ttr.toFixed(2)}, "doucement" ×${doucementCount}, ${clicheHits} cliché(s)`
      : undefined,
  });

  // D7 — Anti-clones (n-gram overlap vs 5 dernières)
  const overlaps = recentStories.slice(0, 5).map((s) => ngramOverlap(story.texte, s.texte, 3));
  const maxOverlap = Math.max(0, ...overlaps);
  dims.push({
    id: 'anti_clones',
    score: maxOverlap > 0.35 ? 1 : maxOverlap > 0.25 ? 2 : 3,
    reason: maxOverlap > 0.25
      ? `Overlap 3-grams ${Math.round(maxOverlap * 100)}% avec une story récente`
      : undefined,
  });

  // D? — Tags TTS (hard fail si tags présents et voice_engine ≠ v3)
  const hasTags = /\[[a-z\s]+\]/i.test(story.texte);
  const tagsAllowed = story.voice_engine === 'eleven_v3';
  dims.push({
    id: 'tags_tts',
    score: hasTags && !tagsAllowed ? 1 : 3,
    reason: hasTags && !tagsAllowed ? 'Tags TTS [..] présents alors que voice_engine ≠ eleven_v3' : undefined,
  });

  // D6 — Cohérence saga (placeholder — vraie logique quand chapitre ≥ 2)
  dims.push({ id: 'coherence_saga', score: 3 });

  const hardFail = dims.some((d) => d.score === 1);
  const softWarnings = dims.filter((d) => d.score === 2 && d.reason).map((d) => d.reason!);
  const rerollPromptHint = dims
    .filter((d) => d.score === 1 && d.reason)
    .map((d) => `- ${d.reason}`)
    .join('\n');

  return { dimensions: dims, hardFail, softWarnings, rerollPromptHint };
}
```

LLM-judge async — calque `callClaude()` mais avec Haiku, temp 0, et JSON strict.

```typescript
// lib/eval/llm-eval.ts
import { z } from 'zod';
import type { BedtimeStory, ChildProfile } from '../parser';
import type { AIConfig } from '../ai-service';
import { LLM_EVAL_SYSTEM_PROMPT, LLM_EVAL_RETRY_PROMPT } from './prompts';
import { anonymizeStoryText } from './rubric-helpers';

export const LlmEvalSchema = z.object({
  rythme: z.number().int().min(0).max(10),
  originalite: z.number().int().min(0).max(10),
  charge_emotionnelle: z.number().int().min(0).max(10),
  fluidite: z.number().int().min(0).max(10),
  justification: z.string().min(1).max(280),
});
export type LlmEvalResponse = z.infer<typeof LlmEvalSchema>;

const HAIKU_MODEL = 'claude-haiku-4-5-20251001';
const FALLBACK_NEUTRAL: LlmEvalResponse = {
  rythme: 5, originalite: 5, charge_emotionnelle: 5, fluidite: 5,
  justification: 'Score neutre — validation JSON échouée, fallback automatique.',
};

export async function evaluateStoryWithLlm(
  story: BedtimeStory,
  child: ChildProfile,
  config: AIConfig,
): Promise<LlmEvalResponse> {
  // Anonymisation OBLIGATOIRE avant envoi (cf. lib/ai-service.ts:333)
  const safeText = anonymizeStoryText(story.texte, child);
  const userMsg = `Tranche d'âge: ${story.tranche_age ?? 'non précisée'}\n\nHistoire:\n${safeText}`;

  const firstAttempt = await callJudge(config, LLM_EVAL_SYSTEM_PROMPT, userMsg);
  const parsed = tryParse(firstAttempt);
  if (parsed.success) return parsed.data;

  // 1 retry max avec prompt durci
  if (__DEV__) console.warn('[eval] JSON invalide, retry avec prompt durci:', parsed.error);
  const second = await callJudge(
    config,
    `${LLM_EVAL_SYSTEM_PROMPT}\n\n${LLM_EVAL_RETRY_PROMPT}`,
    userMsg,
  );
  const reparsed = tryParse(second);
  if (reparsed.success) return reparsed.data;

  if (__DEV__) console.warn('[eval] LLM-judge fallback neutre 5/10');
  return FALLBACK_NEUTRAL; // jamais d'exception remontée — pipeline cosmétique
}

async function callJudge(config: AIConfig, system: string, user: string): Promise<string> {
  const r = await fetch('https://api.anthropic.com/v1/messages', {
    method: 'POST',
    headers: {
      'Content-Type': 'application/json',
      'x-api-key': config.apiKey,
      'anthropic-version': '2023-06-01',
      'anthropic-dangerous-direct-browser-access': 'true',
    },
    body: JSON.stringify({
      model: HAIKU_MODEL,
      max_tokens: 300,
      temperature: 0,
      system,
      messages: [{ role: 'user', content: user }],
    }),
  });
  if (!r.ok) throw new Error(`Haiku ${r.status}`);
  const data = await r.json();
  return data.content?.[0]?.text ?? '';
}

function tryParse(
  raw: string,
): { success: true; data: LlmEvalResponse } | { success: false; error: string } {
  // Strip markdown fences si Haiku en met (observé occasionnellement)
  const cleaned = raw.replace(/^```json\s*|\s*```$/g, '').trim();
  try {
    const json = JSON.parse(cleaned);
    const result = LlmEvalSchema.safeParse(json);
    return result.success
      ? { success: true, data: result.data }
      : { success: false, error: result.error.message };
  } catch (e: any) {
    return { success: false, error: e?.message ?? 'JSON.parse failed' };
  }
}
```

**State Management:**

Aucun état runtime nouveau. Le pipeline est composé de fonctions pures (rubric) et d'un appel HTTP fire-and-forget (LLM-judge). Toute la persistence passe par les couches existantes :

- **Persistence eval** : champs ajoutés au frontmatter `BedtimeStory` (`quality_score: number`, `quality_dimensions: {...}`, `quality_issues: string[]`, `llm_judge: {...} | null`), parsés/sérialisés via `parseStoryFrontmatter` / `serializeBedtimeStory` (`lib/parser.ts:3539, 3591`).

**Context Window Strategy:**

Non-issue dans ce système. Une story FR moyenne fait ~1200-1800 tokens, le system prompt LLM-eval ~500 tokens, total ~2300 tokens d'input. Haiku 4.5 a une fenêtre de 200k tokens — on est à 1.2% d'usage. Pas de troncature, pas de chunking, pas de summarization à prévoir. ## 4b. AI Systems Best Practices

> Written by `gsd-ai-researcher`. Cross-cutting patterns every developer building AI systems needs — independent of framework choice.

### Structured Outputs with Pydantic

TypeScript n'a pas Pydantic ; l'équivalent idiomatique est **Zod** (parsing + inférence statique du type). Pour ce pipeline, le LLM-judge est le seul point qui consomme du JSON LLM-généré, donc c'est le seul endroit où Zod est requis.

```typescript
// lib/eval/llm-eval.ts — schema canonique
import { z } from 'zod';

export const LlmEvalSchema = z.object({
  rythme: z.number().int().min(0).max(10),                  // 0-10 inclus
  originalite: z.number().int().min(0).max(10),
  charge_emotionnelle: z.number().int().min(0).max(10),
  fluidite: z.number().int().min(0).max(10),
  justification: z.string().min(1).max(280),                // tweet-length, FR
});

export type LlmEvalResponse = z.infer<typeof LlmEvalSchema>;
```

**Intégration au flow direct-API** : Anthropic n'a pas de paramètre natif `response_format: json_schema` (contrairement à OpenAI). On force le JSON par instruction stricte dans le system prompt + parsing manuel + Zod validation. Pattern :

1. System prompt : *"Tu DOIS répondre UNIQUEMENT par un objet JSON valide conforme à ce schema : `{ rythme: 0-10, originalite: 0-10, charge_emotionnelle: 0-10, fluidite: 0-10, justification: string ≤ 280 chars }`. Aucun texte avant ou après. Pas de balises markdown."*
2. Strip défensif des fences ` ```json ... ``` ` au cas où Haiku en met.
3. `JSON.parse()` puis `LlmEvalSchema.safeParse()`.
4. Si échec → 1 retry avec prompt durci (`LLM_EVAL_RETRY_PROMPT`) appendant *"Ta réponse précédente n'était pas du JSON valide. Réessaie strictement, sans préambule, sans fences."*
5. Si 2ème échec → fallback `FALLBACK_NEUTRAL` (5/10 partout, justification = "Score neutre — validation JSON échouée"). Loggé en `__DEV__` via `console.warn`. **Jamais d'exception remontée à l'UI** — l'eval LLM est cosmétique, l'app fonctionne sans.

**Retry budget total** : 1 retry. Pas de boucle. Le coût d'un fallback neutre une fois par mois est trivial ; le coût d'une boucle infinie ne l'est pas.

### Async-First Design

Le pipeline a deux phases qui ne doivent PAS partager le même mode d'exécution :

- **Phase 1 (rubric déterministe)** : synchrone pure — `evaluateStoryDeterministic()` retourne en quelques ms. Pas d'`async`, pas de `Promise`. Bloquante par design (la décision de re-roll en dépend).
- **Phase 2 (LLM-judge)** : `async` fire-and-forget. **Jamais `await` sur le chemin critique de génération.** `runEvalPipeline()` retourne `{ rubric, llmJudgeBackground }`, et `hooks/useStories` :
  - lit `rubric` immédiatement (sync),
  - `if (rubric.hardFail && retriesRemaining > 0) → re-roll` (re-passe par phase 1),
  - sauvegarde la story et **affiche l'écran lecture immédiatement**,
  - laisse `llmJudgeBackground` se résoudre en arrière-plan, puis fait un second `vaultManager.writeStory()` quand le score arrive (idempotent).

**Common mistake à éviter** : `const result = await runEvalPipeline(ctx); /* ... */ await result.llmJudgeBackground;` est tentant pour "tout finir avant de naviguer". C'est exactement le pattern qui re-bloque l'UX 1-2s. **Garde-fou** : nommer la variable `llmJudgeBackground` (jamais `llmPromise`) et ajouter un commentaire bloquant `// PHASE 52: NE JAMAIS await — bedtime UX critique`. Une lint rule custom (`no-await-llm-judge`) reste optionnelle pour solo dev.

**Stream vs await** : pas de streaming sur le LLM-judge (réponse JSON courte, pas d'UX à afficher). Pas de streaming sur la génération non plus dans la stack actuelle — la réponse Sonnet est consommée d'un bloc puis parsée JSON. Tout est `await` simple sur la réponse finale.

### Prompt Engineering Discipline

**Séparation system / user — non-négociable** :
- `LLM_EVAL_SYSTEM_PROMPT` : constante module-level dans `lib/eval/prompts.ts`. Contient *toutes* les instructions stables : rôle ("Tu es un évaluateur littéraire d'histoires du soir pour enfants"), critères (4 dimensions avec barème explicite), format (JSON strict avec exemple), garde-fous ("ne juge JAMAIS l'enfant ni les parents — uniquement le texte"). **Jamais reconstruit à chaque appel** — coût zéro compute, mais surtout cache de prompt côté Anthropic potentiellement activable.
- `user message` : minimal — uniquement `Tranche d'âge: X\n\nHistoire:\n<texte anonymisé>`. Rien d'autre. ≤ 100 lignes garanties.

**Few-shot** : inline dans le system prompt — 2 exemples (1 story 🟢 forte avec scores 8-9, 1 story 🔴 faible avec scores 2-4) tirés du golden set, anonymisés. Pas de retrieval dynamique : la diversité des stories est trop faible pour justifier un système de récupération, et 2 exemples calibrent suffisamment Haiku 4.5.

**Token budget** : `max_tokens: 300` est explicite et obligatoire. La réponse cible est ~150-200 tokens (4 entiers + 1 string ≤ 280 chars). Cap à 300 = marge de sécurité contre un Haiku verbeux, **jamais d'output unbounded en production**. Pour la génération existante, `MAX_TOKENS = 512` (lib/ai-service.ts:373) sera étendu si la longue génère des sagas — mais ce n'est PAS dans le scope Phase 52.

**Discipline FR** : tout le pipeline parle français — system prompts, justifications LLM, raisons rubric, frontmatter labels, `console.warn` __DEV__. Aucun anglais user-facing (cf. CLAUDE.md). Les tags techniques internes (id de dimension, clés JSON Zod) restent en anglais snake_case pour cohérence code.

### Context Window Management

**Non-applicable à ce système.** Volumes :

| Source | Tokens approx |
|--------|---------------|
| System prompt LLM-judge (avec 2 few-shot) | ~700 |
| Story FR moyenne (anonymisée) | ~1500 |
| Tranche d'âge + framing | ~30 |
| **Total input par appel judge** | **~2230** |
| Fenêtre Haiku 4.5 | 200 000 |
| **% utilisé** | **~1.1%** |

Aucune stratégie de troncature, summarization, ou compaction nécessaire. Si une saga de chapitre 5+ devait un jour être passée en contexte (pour évaluer la cohérence cross-chapitres), on monterait à ~6-8k tokens — toujours trivial. Ce dossier reste explicitement ouvert dans la dimension `coherence_saga` (Section 4) sans budget context.

### Cost and Latency Budget

**Pricing référence** (Claude Haiku 4.5 — vérifier la doc Anthropic au moment du build, le pricing évolue) : ordre de grandeur ~$1 / MTok input, ~$5 / MTok output.

**Coût par story évaluée** :
- Input : ~2230 tokens × $1/MTok = $0.00223
- Output : ~200 tokens × $5/MTok = $0.00100
- **Total par story : ~$0.0032**

**Coût annuel par famille** (hypothèse 2 stories/jour × 365 jours = 730 stories/an) :
- 730 × $0.0032 = **~$2.34/an/famille**

C'est négligeable et sub-coût d'une seule génération Sonnet. On peut accepter sans hésiter.

**Coût du re-roll** : cap 1 retry × ~$0.02 (Sonnet génération) = ~$0.02 par hard-fail. Si le rubric flag 30% hard-fails sur stock historique baseline, et qu'on suppose 10% en régime stationnaire après calibration prompt, le re-roll ajoute ~$1.5/an/famille. Total Phase 52 : ~$4/an/famille. Trivial.

**Latence** :
- Rubric déterministe : < 50ms (mesurable, n-gram sur 5×1500 tokens reste sub-ms en JS).
- LLM-judge Haiku 4.5 : ~1-2s typique. **Async donc invisible à l'UX bedtime.**
- Re-roll Sonnet : ~3-8s selon longueur. **Visible** mais acceptable car affiché derrière un loader "Une nouvelle version arrive…". UX à designer dans la phase planning.

**Caching** :
- **Pas de caching exact-match utile** : chaque story est unique par construction (date, contexte, mood, memories différents).
- **Anthropic prompt caching** : potentiellement activable sur le `system` du judge (constant) si l'API le supporte au moment du build (vérifier `anthropic-beta: prompt-caching-*` ou équivalent). ROI à 730 calls/an : marginal. **À ne PAS bloquer la phase pour ça.**
- **Idempotence frontmatter** : si `llm_judge` est déjà rempli pour une story, ne PAS rappeler le judge. Économie réelle : 0 (chaque story n'est jugée qu'une fois de toute façon), mais protection contre les bugs de re-trigger.

**Sub-task model routing** : déjà en place — Haiku pour évaluation (lecture critique), Sonnet pour génération (créativité). Ne pas dégrader davantage (Claude Haiku 3 serait moins fiable sur la nuance FR ; on garde 4.5).

**Garde-fou monitoring** : logger en `__DEV__` les durées et les fallbacks neutres :
```typescript
if (__DEV__) console.log(`[eval] judge ${duration}ms, score moyen ${avg}, fallback=${isFallback}`);
```
Pas de télémétrie production (zero-backend). En cas de doute, le parent (single-user) regarde la console Xcode.

| Régression silencieuse rubric | D1-D8 (test golden set) | Détection CI Jest |

| LLM-eval hallucine | Zod sur D9+D10 + fallback neutre | Code fence strip + retry |
| Frontmatter casse parser | Roundtrip test parse → serialize | Test unitaire dédié |
| Coût LLM-eval explose | D9+D10 toujours async, hors chemin critique | Architecture (Section 4) |

**Décision re-roll** : si `dimensions.some(d => d.score === 1 && priority === 'Critical')` → re-roll avec `rerollPromptHint` injecté dans le prompt + `retriesRemaining = 0`. Si 2ème échec, ship la 1ère version + log soft warnings dans frontmatter.

### Eval Tooling

**Primary Tool:** **Jest (déjà installé) + helpers TypeScript purs (`lib/eval/rubric-helpers.ts`).** Aucun outil supplémentaire.

**Justification du non-choix Phoenix / Langfuse / Braintrust** :
Le template par défaut `gsd-eval-planner` recommande Arize Phoenix en self-hosted. **Inapproprié ici** pour 5 raisons cumulatives :

1. **Zero-backend constraint** — l'app est React Native client-only, vault Markdown iCloud, pas de serveur. Phoenix exige un OTLP collector (Python ou Node serveur) que personne n'opère. Solo dev sur TestFlight personnel.
2. **Volume négligeable** — ~2 stories/jour/famille = ~730/an. Aucun signal statistique exploitable par une plateforme dimensionnée pour des milliers de traces/jour.
3. **Persistance déjà naturelle** — chaque story est un fichier Markdown avec frontmatter. Ajouter `quality_score`, `quality_dimensions`, `quality_issues`, `llm_judge` au frontmatter donne un historique queryable par `grep`/parser, sans DB.
4. **Coût opérationnel disproportionné** — installer/maintenir Phoenix vs ouvrir un fichier `.md` dans Obsidian pour voir l'historique d'une story. Le 2ème gagne pour 1 utilisateur.
5. **Bundle size RN** — toute SDK observability serveur ajoute des KB inutiles dans le bundle iOS.

**Source de vérité historique** : frontmatter par story (parsé par `parseStoryFrontmatter` lib/parser.ts:3591). Pour analyse rétrospective : script `scripts/eval-dashboard.ts` (Phase 52, Quick fix complémentaire) qui parse tous les frontmatters stories et imprime distribution flagged + tendances.

### Reference Dataset

**Size:** **20 stories** (golden set hand-rated) → **objectif d'extension à 30** quand le prompt corrigé re-génère le baseline (cible distribution post-Phase 52 : 14 🟢 / 5 🟡 / 1 🔴).

**Composition:**
- 20 fixtures Jest sérialisées en JSON dans `lib/eval/__tests__/fixtures/golden-set.json`, source = `STORIES-NOTABLE.md`
- Distribution baseline pré-Phase 52 : **6 🟢 / 10 🟡 / 4 🔴**
- Couverture critique :
  - **P1 Tags TTS** : 18 stories avec leak (D1 hard fail attendu)
  - **P2 Vocabulaire redondant** : 14 stories (D4 soft/hard)
  - **P3 Pivot saturé** : 18 stories "marche seul" (D8 soft warning)
  - **P4 Entités incohérentes** : 8 stories (D7 — Cagou polysémique)
  - **P5 Titres dupliqués** : 5 stories (D6 hard)
  - **P6 Typos** : 2 stories (D9 LLM-judge)
  - **P7 Quasi-clones** : #15-#18 (D5 n-gram overlap)
- 6 stories "🟢 baseline" pour vérifier qu'on ne flag pas les bonnes (false positive guard)

**Labeling:**
- **Labellisation** : parent (single-user familial) sur les 20 stories, déjà fait dans `STORIES-NOTABLE.md` + `ANALYSIS.md`. Verdict humain par story : `expected: '🟢' | '🟡' | '🔴'` + `expected_dimensions: { D1: 1|2|3, D2: ... }`.

- **Extension** : à chaque hard fail observé en prod (frontmatter `quality_issues` non-vide), parent peut promouvoir la story en fixture (script `scripts/promote-fixture.ts`) avec son verdict expected. Croissance organique du dataset, pas de couverture exhaustive a priori.

## 6. Guardrails

### Online (Real-Time)

Guardrails déterministes exécutés sur le **chemin critique génération**, avant `vaultManager.writeStory()`. Toute violation hard → re-roll (cap 1) avec prompt augmenté. Latence cible totale rubric : < 50ms.

| Guardrail | Trigger | Intervention |
|-----------|---------|--------------|
| **G1 — Tags TTS leakés** (D1) | `voice_engine !== 'eleven_v3'` ET `/\[[a-z\s]+\]/i.test(texte)` | **Strip** post-process (déjà en place commit `30ab12d1`) **+ flag re-roll** si après strip le texte semble tronqué (< target × 0.7) |
| **G2 — Longueur HS** (D2) | `|actual − target| / target > 0.35` | **Re-roll** avec hint `"Cible ${target} mots, ta version fait ${actual}. Ajuste la longueur."` |
| **G3 — Fin non-paisible** (D3) | Dernier ¶ : 0 marqueur apaisant OU ≥ 1 marqueur agitation | **Re-roll** avec hint `"Le dernier paragraphe doit ramener ${child} au sommeil. Pas d'exclamation, pas d'action finale."` |
| **G4 — Titre 100% identique** (D6) | Titre lowercase strip exact match avec un titre des 7 derniers jours du même enfant | **Re-roll** avec hint `"Le titre '${conflictTitle}' a déjà été utilisé le ${date}. Choisis un titre substantiellement différent."` |
| **G5 — N-gram overlap quasi-clone** (D5) | `maxOverlap > 0.35` vs 5 dernières stories du même enfant | **Re-roll** avec hint `"Cette histoire reprend ${pct}% des phrases d'une story récente. Change le ressort narratif et l'incipit."` |
| **G6 — Cap re-roll** | `retriesRemaining === 0` après 1ère tentative ratée | **Ship** la 1ère version + frontmatter `quality_issues: [...]` + log `__DEV__` `console.warn('[eval] re-roll exhausted, shipping with warnings')` |
| **G7 — Idempotence LLM-judge** | `story.frontmatter.llm_judge` déjà rempli | **Skip** appel Haiku — protection contre re-trigger accidentel |

**Hors-scope guardrails online** :
- Vocabulaire D4 → soft warning frontmatter uniquement (re-roll trop coûteux pour bénéfice marginal — le prompt corrigé doit gérer en amont).
- Cohérence entités D7 → soft warning Phase 52 (registry pas encore mature).
- Saturation pivot D8 → soft warning, doit être traité au prompt génération (pas au gating).

### Offline (Flywheel)

| Métrique | Sampling | Action sur dégradation |
|----------|----------|------------------------|
| **F1 — Régression golden set** | **Sur chaque PR touchant `lib/eval/`, `lib/parser.ts`, ou `lib/ai-service.ts`** : `npx jest lib/eval/__tests__/rubric.test.ts` doit reproduire 14/20 flagged à ±1 dim | **Block PR** (CI fail). Investigation : dimension qui a dérivé → réviser seuil ou helper. |
| **F2 — Distribution flagged ratio (mensuel)** | Toutes les stories du mois (volume faible — pas de sampling, exhaustif). Script `scripts/eval-dashboard.ts` parse frontmatters et imprime : `% hard fail, % soft warning, % clean`. | Si **hard fail rate > 15%** sur 30 jours → ouvrir un Quick GSD pour réviser prompt génération (un seuil rubric trop strict OU prompt qui régresse). Si **clean rate < 60%** → idem. |
| **F3 — Re-roll convergence rate** | Toutes les stories avec `quality_retried: true` dans frontmatter | Si **< 50% des re-rolls améliorent** (i.e. la 2ème version ship encore avec issues), réviser `rerollPromptHint` quality. Manuel, mensuel. |
| **F4 — LLM-judge score moyen 30 derniers jours** | Toutes les stories avec `llm_judge` rempli | Si moyenne `(rythme+originalite+charge+fluidite)/4` chute > 1 point vs mois précédent → review manuelle 5 stories échantillonnées + envisager Sonnet en remplacement de Haiku pour le judge. |
| **F5 — Coût LLM-judge cumulé** | Frontmatter `llm_judge.tokens_used` × pricing | Si dépasse $1/mois (= ~300 calls Haiku, signal anomalie volume), investiguer (boucle ? double-call ?). |

**Solo dev note** : pas d'alerting actif (pas de PagerDuty, pas de Slack webhook). Le parent regarde le dashboard manuellement ~1×/mois ou quand une story déçoit. F1 est le seul gate automatique (CI Jest).
