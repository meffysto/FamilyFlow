<objective>
FAM-49 — Offrir un débouché au « trop-plein de feuilles 🍃 » : chaque famille peut configurer
« X feuilles = Y € ». L'enfant appuie sur « Échanger » pour convertir ses feuilles en un crédit
cadeau €. Les feuilles sont déduites immédiatement (miroir PlotUpgradeSheet) et la demande entre
dans une file d'attente parentale (`pending` → `approved` / `rejected`). Un refus rembourse les
feuilles. La feature est OFF par défaut pour ne pas surprendre les familles existantes.

Purpose: transformer une monnaie qui s'accumule sans usage en récompense réelle, validée par le parent.
Output: config famille (toggle + ratio), file d'attente persistée dans le vault, UI enfant + UI parent.
</objective>

# Décisions de persistance (résolues par le planner)

PERSISTANCE DES CADEAUX EN ATTENTE → **nouveau fichier vault markdown** `cadeaux-en-attente.md`
(à la racine du vault, SANS préfixe de dossier — cohérent avec la convention `gami-{id}.md`,
voir décision [Phase 26-02]). Une paire `parsePendingGifts` / `serializePendingGifts` dans
`lib/parser.ts`.

Justification du choix vault-markdown plutôt que SecureStore :
- Le vault est la source de vérité des données familiales ; une file d'attente de demandes
  enfant→parent EST une donnée familiale (pas une préférence d'appareil).
- Survit au relaunch ET se synchronise via iCloud (visible pour le parent).
- SecureStore est conçu pour des prefs courtes, pas pour une liste de records qui grandit.
- Réutilise le pattern paire parse*/serialize* déjà établi (cf. CLAUDE.md, lib/parser.ts).

LE RATIO + LE TOGGLE → restent dans `GamificationConfig` (SecureStore, clé `gami_config_v1`),
comme demandé, via `loadGamiConfig` / `saveGamiConfig`.

CACHE_VERSION → **PAS de bump nécessaire**. `cadeaux-en-attente.md` n'est PAS un domaine caché
(comme les stories en Phase 52). `GamificationConfig` vit dans SecureStore, pas dans `vault-cache`.
Aucun type caché (Task, Profile base, etc.) ne change.

POLITIQUE DE REMBOURSEMENT (résolue) : **refus = remboursement des feuilles à l'enfant** (défaut
raisonnable de la décision LOCKED). `approved` ne rembourse jamais (feuilles déjà dépensées).
